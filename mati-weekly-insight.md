# 🤖 mati.a 주간 자동 인사이트 — 작업 지시서

> 이 파일은 mati.a(또는 예약 에이전트)가 **매주 1회** 보고 그대로 실행하는 "업무 지시서"입니다.
> 사람이 읽어도 무슨 일을 하는지 알 수 있게 적었습니다.

## 한 줄 목적
지운 업무일지에 쌓인 일지(회고·아이디어·할일·주간회고)를 매주 읽어, **패턴·연결·놓친 것·돌아보는 질문**을 담은 인사이트 한 편을 만들어 앱의 ✨인사이트 탭에 자동 저장한다. (옵션) Slack `#지운-업무일지`에 요약을 공유한다.

## 언제 실행하나
- **매주 1회**, 월요일 아침 권장 (지난 한 주를 돌아보는 타이밍).
- 직전 실행 이후 **새 일지가 없으면** 인사이트를 만들지 말고 건너뛴다(중복 방지).

---

## 데이터 위치 & 형식

- **저장소(비공개):** `company-dari/jiwoon-worklog-data`
- **파일:** `data.json` (앱이 읽고 쓰는 단일 데이터 파일)
- **형식:**
```jsonc
{
  "cats":  [ {"name":"마케팅","color":"#8b5cf6"}, ... ],   // 카테고리
  "cards": [ {"text":"...","col":"todo|doing|done","pri":true,"cat":"마케팅"}, ... ], // 칸반 할일
  "days":  { "2026-06-26": {"retro":"오늘 회고...","idea":"아이디어..."}, ... },        // 일일 기록
  "weeks": { "2026-W26": {"good":"...","bad":"...","learn":"...","next":"..."}, ... },  // 주간 회고
  "insights": [ {"when":"6/26","text":"인사이트 본문...","auto":true,"by":"mati.a"}, ... ] // ← 여기에 추가
}
```
- 앱은 인사이트를 표시할 때 `when`(날짜 라벨)과 `text`(본문)만 사용한다. `auto`/`by`는 출처 표시용(있어도 무시되니 자유롭게 넣어도 됨).
- 인사이트 목록은 **최신이 맨 앞**(배열 앞쪽 unshift).

## 실행 단계 (에이전트용)

### 1) data.json 읽기
GitHub API로 현재 파일을 받는다. `sha`는 쓸 때 필요하니 꼭 보관한다.
```
GET https://api.github.com/repos/company-dari/jiwoon-worklog-data/contents/data.json
Authorization: Bearer <GITHUB_TOKEN>
Accept: application/vnd.github+json
```
응답의 `content`(base64)를 디코드 → JSON 파싱. 응답의 `sha`를 기억.

### 2) 분석할 게 있는지 확인
- `days`·`weeks`·`cards`에 내용이 있는지 본다.
- 직전 인사이트(`insights[0].when`) 이후 **새로 작성된 일지가 없다면 종료**(아무것도 쓰지 않음).

### 3) 일지를 사람이 읽는 글로 정리
아래 순서로 텍스트를 만든다 (앱의 내보내기 형식과 동일):
1. 🗂️ 업무 보드 (col별: 해야 함/진행 중/완료, `❗`=우선순위, `#카테고리`)
2. 📅 일일 기록 (날짜별 회고/아이디어, 최신순)
3. 🔁 주간 회고 (주별 잘된 것/아쉬운 것/배운 것/다음 주)

### 4) 분석 (이 지시문대로 생각한다)
> 다음은 지운(사용자)의 업무 일지다. 한 주를 돌아보는 코치 입장에서, 따뜻하지만 솔직하게 분석해라. 한국어, 반말 금지(존댓말), 군더더기 없이.
> 1. **반복되는 패턴·주제** — 어떤 카테고리(마케팅/재무/거래처/운영/개인)에 쏠렸는지, 자주 미루는 일, 에너지가 어디로 갔는지.
> 2. **연결되는 인사이트** — 떨어진 날짜의 생각·아이디어를 엮어, 혼자 보면 안 보이는 흐름을 짚어준다.
> 3. **놓치고 있는 영역** — 거의 안 적힌 카테고리, 계속 미뤄지는 일, 신경 못 쓰는 부분.
> 4. **돌아보는 질문 2~3개** — 스스로 더 깊이 생각하게 만드는 질문.
> 분량은 한 화면에 읽을 정도(과하지 않게). 데이터가 적으면 "아직 데이터가 적어 단정은 어렵지만…"으로 시작하고 짧게.

### 5) 인사이트 객체 만들기
```json
{ "when": "<오늘 M/D, 예: 6/26>", "text": "<4번 분석 결과 전체>", "auto": true, "by": "mati.a" }
```

### 6) data.json에 저장(쓰기)
- 1)에서 읽은 JSON의 `insights` 배열 **맨 앞에** 새 인사이트를 넣는다(`unshift`). **기존 데이터(cards/days/weeks/insights)는 절대 지우지 않는다.**
- 전체 JSON을 다시 base64로 인코딩해 PUT.
```
PUT https://api.github.com/repos/company-dari/jiwoon-worklog-data/contents/data.json
body: { "message":"weekly insight <날짜>", "content":"<base64>", "sha":"<1)에서 받은 sha>" }
```
- 만약 409/422(충돌)면: 1)을 다시 GET → 최신 sha·내용으로 인사이트 다시 넣고 PUT 재시도.

### 7) (옵션) Slack 공유
`#지운-업무일지`에 인사이트 앞부분 2~3줄 + "전체는 앱 ✨인사이트 탭에서 보세요" 정도로 짧게 공유.

## 주의사항 (중요)
- **기존 데이터 보존**: 반드시 읽은 JSON을 수정해서 쓴다. 새 JSON으로 통째 덮어쓰지 않는다.
- **중복 방지**: 새 일지가 없으면 인사이트를 만들지 않는다.
- **토큰**: `jiwoon-worklog-data` 저장소의 contents 읽기/쓰기 권한이 있는 토큰 필요(비공개 저장소).
- **앱 확인**: 사용자가 앱에서 ☁️ → "지금 동기화"를 누르면 새 인사이트가 ✨인사이트 탭에 나타난다.
